run it yourself · local-first by default

Self-host in four commands.

project-mri is self-hosted by default — the whole experience runs on your own machine or server. Install, initialize, serve, scan. No account, no telemetry, no cloud dependency.

/// the walkthrough

Four steps, start to finish.

01
Install

One Python package. The dashboard ships as pre-built static assets, so there is no Node runtime to install.

02
Initialize

Creates your admin user and a local database. Prompts for a username, a password (8+ chars), and a config path. Idempotent — safe to re-run.

03
Serve

Binds to 127.0.0.1:7331 by default — local-only, no auth needed on loopback. Then open the dashboard in your browser.

04
Scan a repository

Point it at a local directory or a git URL. A remote URL is shallow-cloned into a sandbox and cleaned up afterward.

quickstart.shfour commands
# 1 · install (pipx, pip, or uv all work)
$ pipx install project-mri
$ mri --version
project-mri 0.1.0

# 2 · initialize — admin user + local database
$ mri init

# 3 · serve on loopback, then open the dashboard
$ mri serve
# → http://localhost:7331/dashboard/

# 4 · scan a local checkout…
$ mri scan /path/to/your/code
# …or a remote repo (shallow clone, auto-cleanup)
$ mri scan https://github.com/yourorg/yourrepo.git
/// the local-first guarantee

Proven, not asserted.

// your code never leaves your machine

No SaaS, no account, no phone-home. The only network access is what you explicitly ask for — cloning a repo URL you named, or a webhook you configured.

// zero telemetry, proven not asserted

A test replaces the socket layer and fails the build if any code path opens a non-loopback connection. The suite runs a full local scan and a full session-log ingest with the network sealed.

// fail-closed when exposed

Bind to a public interface without configuring auth and the server refuses to start, rather than serving in the open. Loopback stays frictionless; everything else needs an API key or a dashboard user.

/// exposing it on a network

Reachable from another host.

To reach the dashboard from another host, bind to all interfaces and put it behind a reverse proxy. Because the server fails closed on a non-loopback bind, you must configure auth first — set an API key or create a dashboard user with mri init.

expose.shauth required
# requires auth to be configured, or it refuses to start
$ MRI_HOST=0.0.0.0 mri serve
