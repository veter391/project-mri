01 · git history analyzer

Mines the full commit history of the repository. Per-file metrics that surface where the risk actually lives — not where someone guessed it does.

What it extracts

  • Churn — total lines added/removed over 30/90/365-day windows per file
  • Change frequency — how often each file is touched
  • Ownership evolution — who owned a file last month vs. last year
  • Co-change patterns — files that move together (high blast radius signal)
  • Hotspots — files with high recent activity + high complexity
  • Optional AI-commit flagging — heuristic detection of generic large commits

02 · code structure analyzer

Parses the current codebase with tree-sitter to extract a real Abstract Syntax Tree — not regex guessing. Multi-language out of the box.

Supported languages

python typescript javascript go rust java php c / c++

What it extracts

  • Module / file structure with package boundaries
  • Imports & dependencies between files (for the dependency graph)
  • Function / method definitions & calls (for the call graph)
  • Complexity metrics — cyclomatic, cognitive, Halstead
  • Dead code & duplication signals (initial versions)
  • Public API surface — for documentation auto-generation
// why tree-sitter (not regex)
Regex breaks on Python decorators, Go generics, TypeScript satisfies clauses. tree-sitter gives you a real AST so we never get confused by syntax. Parsing is incremental — fast even on 100k+ files.
03 · graph builder

Combines data from Git history and code structure into four interconnected graphs. Each one answers a different question.

dependency graph
"what depends on what" — files/modules connected by imports. Used to find cycles, dead branches, blast-radius.
call graph
"what calls what" — function/method-level edges. Used to find unreachable code and refactor targets.
evolution graph
"how did the structure change" — temporal layers of the above. Used to detect coupling drift and architecture erosion.
knowledge graph
"who knows what" — files ↔ contributors ↔ recent changes. Used to compute bus factor and find knowledge islands.
decision graph
[Phase 3] links commits, PRs, issues, and architectural decisions. The "why" behind every refactor.

Graphs are built in-memory with NetworkX and persisted to SQLite/DuckDB for fast queries. Visualizations use Cytoscape.js (interactive) or Pyvis (static HTML export).

04 · scoring & risk engine

Composite, explainable scores. Every number links to the signals behind it. Click any score → see the contributing factors → drill into the files.

Core scores (MVP)

// explainability principle
We will never ship a score you can't decompose. If architecture_health = 71, you'll see: complexity +18, coupling -12, modularity +25, etc. Drill down. Always.
05 · ai impact detector

Detects and quantifies the influence of AI coding tools on your codebase. Two layers: heuristic (always on) + optional LLM (Ollama).

Heuristic signals

  • Large commits with generic messages ("Update", "Fix", "Refactor")
  • Files modified in bursts with no intermediate commits
  • Sudden spikes in authorship (one person, many files, same hour)
  • Comments and code style homogeneity across recent changes

Optional: session log ingestion

If you point MRI at AI tool logs (~/.claude/, ~/.cursor/), it can compute exact attribution: which files were touched by which sessions, which prompts led to which commits. Phase 3: MCP server for live queries.

// privacy
AI log ingestion is fully opt-in. Default mode uses commit metadata only. No prompt content is read unless you explicitly enable it.
