Four principles that decide every design decision in Project MRI. Read them. Question them. Hold us to them.

01
Facts over magic scores.

A composite score is only useful if you can decompose it. If architecture_health = 71, you should see why — not a black box, but a tree of named, traceable signals. We will never ship a number you can't audit.

Most existing tools give you a score and trust you to react. We give you the score AND the receipts. Every signal is a real, queryable, exportable piece of evidence.

02
Local-first. Always.

Your code is yours. We will never phone home. We will never require a login. We will never bundle analytics, telemetry, or "usage data" by default. The local database is a regular file on your disk — readable, exportable, deletable.

Cloud features (managed instance, team sync, hosted dashboard) will be opt-in, transparently priced, and never required for core functionality. GDPR compliance isn't a feature — it's the baseline.

03
Explain before recommend.

Most tools lead with recommendations: "refactor this", "fix that", "invest here". We lead with data. The report shows the architecture, the trends, the risk signals — and only then offers suggestions. You should be able to disagree with the recommendation and still understand the analysis.

This is the opposite of "AI assistant that just tells you what to do". We trust the operator to make decisions. We make sure they have the facts.

04
Open source core. Forever.

The decision engine, the analyzers, the scoring algorithms, the report generators — all open source. MIT-licensed. Fork it. Audit it. Ship it inside your company. Build a competitor on top of it. We want this.

Commercial features (managed hosting, enterprise SSO, support contracts) live on top of the open core, never instead of it. The MIT license will not change without explicit community consent.

/// non-goals

What we won't build.

Every project needs boundaries. These are ours — for now, and probably for the foreseeable future.

  • Real-time monitoring / runtime observability. We scan repos, not live systems. Use Prometheus, Datadog, or OpenTelemetry for that.
  • Security vulnerability scanning. Trivy, Snyk, and GitHub's own Dependabot do this better. We won't reinvent.
  • Replacing SonarQube. Different focus. Sonar measures code style and security; MRI measures architecture and history.
  • Automatic code refactoring. We tell you what's risky. We never silently modify your code.
  • Cloud-hosted SaaS by default. The core stays local. Cloud is opt-in.
  • Replacing your Git host. We don't clone your code to our servers. We read your local .git directory.
/// why open source

The license is part of the product.

Not a marketing line. A structural commitment.

auditabilitywhy

If a tool tells you your codebase has a problem, you should be able to read the algorithm that decided that. Closed-source analysis tools are unfalsifiable claims.

trustwhy

Codebases are the most sensitive artifact an engineering team owns. "Trust us, we run it locally" is weaker than "read the source and verify".

longevitywhy

If we disappear tomorrow, the tool still works. The community can fork. The MIT license guarantees no rug-pull.

ecosystemwhy

Plugins, integrations, language analyzers, scoring models — they emerge faster from a community than from a roadmap. We want the long tail.
