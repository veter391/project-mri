roadmap · phases

From foundation to ecosystem.

Four phases, each shippable on its own. We don't hold features hostage to a future release. Every phase delivers real value to a real user.

// prioritization framework

Every feature gets scored on five criteria before it ships. If it doesn't clear all five, it waits.

  1. Real painful problem. Is this something users actually struggle with, or a feature we'd just like to have?
  2. Deliverable value in this phase. Can we ship something meaningful now, or is it dependent on infrastructure that doesn't exist yet?
  3. Local-first & transparent. Does this stay aligned with our manifesto? No exceptions.
  4. Technically feasible. Can we build it with the team and tools we have today?
  5. Attracts contributors. Will this make the project more welcoming or more intimidating?

phase 1 · core mvp
The first useful version.
estimated · 4–8 weeks
First version that provides real diagnostic value. Both a quick CLI summary and a nice HTML report with graphs. Enough that a tech lead could use it in a 1:1 to discuss architecture.
  • Dependency & call graph extraction (tree-sitter, 6+ languages)
  • Churn, hotspot, ownership analysis (with time-series)
  • Composite scoring engine (architecture health, tech debt, bus factor, knowledge islands)
  • Self-contained HTML report with interactive graph viz (Pyvis → Cytoscape.js)
  • Local SQLite cache for incremental re-runs
  • Markdown + HTML report export
  • Improved CLI with progress bars + JSON export
// success criteria
Analysis completes in < 60s on a 50k-line Python project. The HTML report opens offline and links every score to its contributing signals.
phase 2 · intelligence
Daily-driver tool.

Make MRI a tool people actually open every day. Rich desktop app, historical trends, and the AI Impact detector that no competitor has.
  • Historical evolution analysis (coupling changes, hotspot trends over time)

phase 3 · ecosystem
A platform, not a tool.
future
Become the standard way codebases are queried — by humans and by AI agents. Plugin architecture. Decision provenance. Multi-repo workspaces.
  • Decision provenance graph (commits ↔ PRs ↔ issues ↔ AI sessions)
  • MCP server for Claude Code, Cursor, Aider, etc.
  • Plugin architecture for community analyzers + scoring models
  • Multi-repository support & comparison
  • Optional opt-in anonymized insights sharing
  • Possibly: hosted managed instance for teams (paid, core stays free)
// success criteria
AI agents routinely query MRI as part of their context. Community plugins exist for niche languages and scoring frameworks. The tool is referenced as an industry standard.
/// explicit non-goals

Things we will not build (intentionally).

Saying no is part of building something great. Other tools cover these better — use them.

real-time observability

MRI scans repos, not live systems. Use Prometheus, Datadog, OpenTelemetry, or Sentry for runtime monitoring.

security vuln scanning

Trivy, Snyk, Dependabot, GitHub Advanced Security — all do this well. Not our wheelhouse.

replacing SonarQube

Sonar measures style + security. MRI measures architecture + history. Complementary, not competitive.

automatic refactoring

We never silently modify your code. We tell you what's risky. You decide.

/// long-term

Where this is going.

Project MRI started because every tool that existed in this space asked us to upload our code. That's not how engineering teams work. That's not how trust works. The 5-year-old MRI report in your wiki should still be readable. The score should still mean what it meant in 2025.

We want MRI to become the standard way engineering teams understand their own codebases — the way git log is the standard way you understand history, the way grep is the standard way you search. A tool so fundamental you forget it was ever new.

We're not building a product. We're building infrastructure for the next decade of software engineering.
