architecture · how it works

From local repo to actionable report.

Seven documented steps. Modular by design — every analyzer is independent. Skip what you don't need. Replace what doesn't fit your workflow.

/// pipeline overview

The 5-layer architecture.

Input flows through the orchestrator → parallel analyzers → storage → output. Each box is an independent module.

input

Local Git repo (or URL to clone). Optional: AI session logs, GitHub Issues/PRs.

orchestrator

Coordinates analyzers, handles config, runs in parallel where possible, reports progress.

5 analyzers

Git history, code structure, graph builder, scoring engine, AI impact. Each is a standalone module.

storage

SQLite for structured metadata. DuckDB for fast analytical queries on large result sets.

output

CLI summary · self-contained HTML · interactive dashboard (Phase 2) · JSON + MCP (Phase 3).

extensibility

Add your own analyzer as a Python module. Plug into the orchestrator. Output formatters are pluggable too.

/// pipeline detail

The 7 steps of a single MRI scan.

Each step is idempotent and incremental. Re-running skips work that's already cached.

  1. Repository ingestion. User provides path to a local Git repo. MRI validates it, loads config (languages, history depth), and creates a fresh SQLite cache. Optional: clone from URL.
  2. Git history analysis. Walk the full commit log. Per-file churn (lines added/removed), change frequency, ownership evolution, co-change patterns, hotspots. Optionally flag large generic commits.
  3. Code structure analysis. Run tree-sitter on every source file. Extract AST, imports, call sites, complexity metrics. Multi-language. Incremental — only re-parse changed files.
  4. Graph construction & enrichment. Build dependency graph, call graph, evolution graph, knowledge graph. Combine Git data (who changed what) with code data (what depends on what). Persist + index for fast queries.
  5. Scoring & risk analysis. Compute explainable composite scores. Every score is a function of named, traceable signals. Drill-down to contributing factors, then to specific files.
  6. Optional AI impact analysis. If enabled, ingest AI session logs and attribute changes to specific sessions/prompts. Generate an AI Influence Report: concentration of AI changes, risk patterns, recommendations.
  7. Output generation. CLI summary (always), HTML report (always), interactive dashboard (Phase 2), JSON for AI agents (Phase 3). Self-contained means no external requests at render time.
/// components

Seven core components.

Each component is independently testable. You can swap any of them.

1 · analysis orchestrator

The conductor. Validates input, loads config, schedules analyzers, aggregates results, reports progress. Stateless — every run is reproducible.

  • Parallel execution where safe
  • Resume from cache on re-run
  • Clean separation of concerns
2 · git history analyzer

Wraps libgit2 (via pygit2) or shells out to native git. Computes churn with rolling windows, ownership timelines, co-change matrix.

  • Streaming parse (handles huge histories)
  • Heuristic AI-commit detection
  • Time-series storage for trends
3 · code structure analyzer

tree-sitter bindings for each supported language. Builds in-memory graphs (NetworkX) of imports and calls. Computes complexity metrics.

  • 6+ languages supported
  • Incremental re-parse
  • Public API surface extraction
4 · graph builder

Merges Git and code-structure outputs into four graph types. Persists to SQLite for query speed, exports to JSON/GraphML for portability.

  • NetworkX for analysis
  • Cytoscape.js for visualization
  • Pre-computed shortest paths
5 · scoring & risk engine

Pure functions over the graph + metrics. Each score returns a number AND its contributing factors. No hidden weights.

  • 6 core scores in MVP
  • Custom scoring via plugin
  • Threshold-based risk tiers
6 · ai impact detector

Two modes. Always-on: commit-pattern heuristics. Opt-in: ingest AI session logs for exact attribution. Optional local LLM (Ollama) for natural-language commit classification.

  • Local LLM, no cloud
  • Session-level attribution
  • Trend detection (rising AI share?)
7 · output layer

Pluggable formatters. The data model is JSON-stable; the visualization can be CLI, HTML, or interactive. Phase 3 adds MCP for agent access.

  • CLI (Typer + Rich)
  • HTML (Jinja2 templates)
  • Tauri dashboard (Phase 2)
  • MCP server (Phase 3)
/// principles

Five non-negotiable design rules.

01 · local-firstprivacy

Zero data leaves the machine by default. No telemetry. No analytics. The local database is just a file on your disk. Cloud features are always opt-in.

02 · transparencyexplainability

Every score and every graph edge must be explainable. If you can't trace a number back to its contributing signals, it doesn't ship.

03 · modularityextensibility

Analyzers are independent Python modules. New languages, new metrics, new visualizers — all pluggable. Community-driven scoring models welcome.

04 · performancescalability

Should handle repositories with 100k+ files and 10+ years of history on a developer laptop in under a minute. Incremental by default. DuckDB for heavy queries.

05 · ai-friendlyagent-native

JSON-stable schema. MCP server (Phase 3) so AI agents can query MRI live. AI Impact detector so you know what your AI tools are actually doing.

06 · timelesslongevity

Useful even 5–10 years from now, independent of specific AI models. We measure your codebase, not the LLM-of-the-month.
